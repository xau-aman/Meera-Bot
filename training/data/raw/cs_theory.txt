<|user|>What is an operating system?<|end|>
<|meera|>An operating system is the software layer between you and the hardware. It manages:
- Process management (running programs)
- Memory management (RAM allocation)
- File system (storing/retrieving data)
- I/O management (keyboard, disk, network)
- Security & access control

Think of it as a restaurant manager — it decides who gets served (CPU time), where they sit (memory), and handles complaints (errors).

Examples: Linux, Windows, macOS. For servers, Linux dominates. Know Linux basics — interviewers love it.<|end|>
<|user|>Explain process vs thread.<|end|>
<|meera|>**Process**: an independent program in execution. Has its own memory space, file handles, and resources. Processes are isolated — one crashing doesn't kill others.

**Thread**: a lightweight unit of execution within a process. Threads share the same memory space but have their own stack.

| Feature | Process | Thread |
|---------|---------|--------|
| Memory | Separate | Shared |
| Creation | Heavy (fork) | Light |
| Communication | IPC (pipes, sockets) | Direct (shared memory) |
| Crash impact | Isolated | Can crash whole process |

Example: Chrome uses multiple processes (one per tab). Each tab might use multiple threads (rendering, JS, network).

Interview favorite: "Why use threads over processes?" — lower overhead, shared memory makes communication easy. But shared memory = race conditions. That's where synchronization comes in.<|end|>
<|user|>What is deadlock?<|end|>
<|meera|>Deadlock is when two or more processes are stuck forever, each waiting for a resource the other holds.

Four conditions (ALL must be true):
1. **Mutual Exclusion** — resource can't be shared
2. **Hold and Wait** — holding one resource, waiting for another
3. **No Preemption** — can't forcibly take a resource
4. **Circular Wait** — A waits for B, B waits for A

Classic example: two people in a narrow hallway, both refusing to step aside.

Prevention — break any one condition:
- Order resource requests (breaks circular wait)
- Request all resources at once (breaks hold and wait)
- Allow preemption (breaks no preemption)

Detection: build a resource allocation graph, check for cycles.

This comes up in almost every OS interview. Know the four conditions cold.<|end|>
<|user|>Explain virtual memory.<|end|>
<|meera|>Virtual memory creates an illusion that each process has its own large, contiguous memory space — even if physical RAM is limited.

How it works:
1. Each process gets a virtual address space
2. Virtual addresses are mapped to physical addresses via page table
3. Only active pages stay in RAM, rest go to disk (swap space)

**Paging**: memory divided into fixed-size pages (typically 4KB). When a page isn't in RAM → page fault → OS loads it from disk.

**TLB (Translation Lookaside Buffer)**: cache for page table entries. Makes address translation fast.

Benefits:
- Process isolation (security)
- More processes than physical RAM allows
- No external fragmentation

The cost? Page faults are expensive (disk is ~1000x slower than RAM). That's why thrashing happens — too many page faults, system grinds to a halt.<|end|>
<|user|>What is DBMS and why do we need it?<|end|>
<|meera|>A Database Management System (DBMS) is software that stores, retrieves, and manages data efficiently.

Why not just use files?
- **Data redundancy** — files duplicate data everywhere
- **Inconsistency** — update one file, forget another
- **No concurrent access** — two users editing same file = chaos
- **No security** — anyone can read/write files
- **No query language** — finding data in files is painful

DBMS solves all of this with:
- Structured storage (tables, schemas)
- SQL for querying
- ACID transactions
- Access control
- Concurrent access handling

Types:
- Relational (PostgreSQL, MySQL) — tables with relationships
- NoSQL (MongoDB, Redis) — flexible schemas
- Graph (Neo4j) — relationship-heavy data

For interviews: know relational DBMS deeply. SQL, normalization, indexing, transactions — these are must-knows.<|end|>
<|user|>Explain ACID properties.<|end|>
<|meera|>ACID ensures database transactions are reliable:

**Atomicity** — all or nothing. If a transaction has 5 steps and step 3 fails, all 5 are rolled back. Like a bank transfer — debit AND credit must both happen, or neither.

**Consistency** — database moves from one valid state to another. All constraints, triggers, and rules are satisfied.

**Isolation** — concurrent transactions don't interfere. Transaction A can't see Transaction B's uncommitted changes.

**Durability** — once committed, data survives crashes. Written to disk, not just RAM.

Example: transferring ₹500 from Account A to B:
1. Debit A by ₹500
2. Credit B by ₹500

Atomicity: both happen or neither. Consistency: total money stays same. Isolation: another transaction sees either before or after, not in-between. Durability: power goes out after commit? Data is safe.

This is interview gold. Know it with examples.<|end|>
<|user|>What is normalization in databases?<|end|>
<|meera|>Normalization is organizing tables to reduce redundancy and dependency.

**1NF** — no repeating groups, atomic values only.
Bad: `hobbies = "reading, coding"` → Good: separate rows.

**2NF** — 1NF + no partial dependencies. Every non-key column depends on the WHOLE primary key.

**3NF** — 2NF + no transitive dependencies. Non-key columns don't depend on other non-key columns.

**BCNF** — stricter 3NF. Every determinant is a candidate key.

Example:
```
Students(id, name, dept_id, dept_name)
```
Problem: dept_name depends on dept_id, not on student id (transitive dependency).

Fix (3NF):
```
Students(id, name, dept_id)
Departments(dept_id, dept_name)
```

Denormalization is the opposite — intentionally adding redundancy for read performance. Used in data warehouses and analytics.

Interview tip: know up to 3NF with examples. That's usually enough.<|end|>
<|user|>Explain indexing in databases.<|end|>
<|meera|>An index is like a book's index — instead of reading every page, you jump directly to what you need.

Without index: full table scan — O(n). With index: O(log n) using B-tree.

```sql
CREATE INDEX idx_email ON users(email);
-- Now: SELECT * FROM users WHERE email = 'x' is fast
```

Types:
- **B-tree index** — default, good for range queries and equality
- **Hash index** — O(1) for equality, no range support
- **Composite index** — multiple columns: `(last_name, first_name)`
- **Unique index** — enforces uniqueness

Trade-offs:
- Faster reads, slower writes (index must be updated on INSERT/UPDATE)
- Extra storage space
- Too many indexes = diminishing returns

Rule of thumb: index columns you frequently filter (WHERE), join (JOIN), or sort (ORDER BY) on. Don't index everything blindly.<|end|>
<|user|>What is TCP vs UDP?<|end|>
<|meera|>Two transport layer protocols with very different philosophies:

**TCP (Transmission Control Protocol)** — reliable, ordered delivery.
- 3-way handshake (SYN → SYN-ACK → ACK)
- Guarantees delivery (retransmits lost packets)
- Ordered (packets arrive in sequence)
- Flow control & congestion control
- Use: HTTP, email, file transfer — anything where data must be complete

**UDP (User Datagram Protocol)** — fast, fire-and-forget.
- No handshake, no connection
- No delivery guarantee
- No ordering
- Minimal overhead
- Use: video streaming, gaming, DNS, VoIP — where speed > reliability

| Feature | TCP | UDP |
|---------|-----|-----|
| Reliable | Yes | No |
| Ordered | Yes | No |
| Speed | Slower | Faster |
| Overhead | High | Low |
| Connection | Yes | No |

Simple analogy: TCP is a phone call (connection, confirmation). UDP is shouting across the room (fast, no guarantee they heard you).<|end|>
<|user|>Explain HTTP and HTTPS.<|end|>
<|meera|>**HTTP** (HyperText Transfer Protocol) — the language browsers and servers speak. It's stateless — each request is independent.

Request structure:
```
GET /api/users HTTP/1.1
Host: example.com
```

Response:
```
HTTP/1.1 200 OK
Content-Type: application/json
{"users": [...]}
```

Methods: GET (read), POST (create), PUT (update), DELETE (remove), PATCH (partial update).

Status codes:
- 2xx: success (200 OK, 201 Created)
- 3xx: redirect (301, 304)
- 4xx: client error (400 Bad Request, 401 Unauthorized, 404 Not Found)
- 5xx: server error (500 Internal Server Error)

**HTTPS** = HTTP + TLS/SSL encryption. Data is encrypted in transit — no one can sniff your passwords.

How TLS works (simplified):
1. Client hello → server sends certificate
2. Client verifies certificate
3. Both agree on encryption key
4. All data encrypted from here

Always use HTTPS. No exceptions.<|end|>
<|user|>What are the four pillars of OOP?<|end|>
<|meera|>The four pillars of Object-Oriented Programming:

**1. Encapsulation** — bundle data + methods together, hide internal details.
```python
class BankAccount:
    def __init__(self):
        self.__balance = 0  # private
    def deposit(self, amount):
        self.__balance += amount
    def get_balance(self):
        return self.__balance
```

**2. Abstraction** — show only what's necessary, hide complexity.
You call `list.sort()` without knowing the sorting algorithm inside.

**3. Inheritance** — create new classes from existing ones.
```python
class Animal:
    def speak(self): pass

class Dog(Animal):
    def speak(self): return "Woof!"
```

**4. Polymorphism** — same interface, different behavior.
```python
animals = [Dog(), Cat()]
for a in animals:
    a.speak()  # each speaks differently
```

Interview tip: don't just define them — give real-world examples. Encapsulation = ATM (you interact with buttons, not internal circuits). Inheritance = "is-a" relationship. Polymorphism = one remote, many devices.<|end|>
<|user|>Explain SOLID principles.<|end|>
<|meera|>SOLID — five design principles for clean, maintainable code:

**S — Single Responsibility**: a class should have one reason to change.
Bad: `UserService` handles auth, email, and database. Good: split into `AuthService`, `EmailService`, `UserRepository`.

**O — Open/Closed**: open for extension, closed for modification.
Add new behavior by extending, not changing existing code. Use interfaces/abstract classes.

**L — Liskov Substitution**: subclass should be usable wherever parent class is expected.
If `Bird` has `fly()`, `Penguin extends Bird` breaks this. Fix: don't make Penguin a Bird, or remove fly from Bird.

**I — Interface Segregation**: don't force classes to implement methods they don't use.
Bad: one fat `Worker` interface with `code()`, `test()`, `manage()`. Good: separate `Coder`, `Tester`, `Manager` interfaces.

**D — Dependency Inversion**: depend on abstractions, not concrete implementations.
Bad: `OrderService` directly creates `MySQLDatabase`. Good: `OrderService` depends on `Database` interface.

These aren't just theory — they make your code testable, flexible, and maintainable. Interviewers love asking about these with real examples.<|end|>
<|user|>What is DNS and how does it work?<|end|>
<|meera|>DNS (Domain Name System) translates human-readable domain names to IP addresses. It's the internet's phone book.

When you type `google.com`:
1. Browser checks local cache
2. OS checks its cache
3. Query goes to recursive resolver (usually your ISP)
4. Resolver asks root nameserver → "Who handles .com?"
5. Root says → ask .com TLD nameserver
6. TLD says → ask Google's authoritative nameserver
7. Authoritative nameserver returns IP: 142.250.x.x
8. Browser connects to that IP

Record types:
- A: domain → IPv4
- AAAA: domain → IPv6
- CNAME: alias → another domain
- MX: mail server
- NS: nameserver
- TXT: arbitrary text (used for verification)

TTL (Time To Live): how long to cache the result.

Fun fact: DNS is one of the most common interview topics for system design. "Design a URL shortener" always involves DNS discussion.<|end|>
<|user|>What is REST API?<|end|>
<|meera|>REST (Representational State Transfer) is an architectural style for designing APIs. Most web APIs follow REST.

Principles:
- **Stateless** — each request contains all info needed
- **Resource-based** — everything is a resource with a URL
- **HTTP methods** — map to CRUD operations
- **Uniform interface** — consistent URL patterns

```
GET    /api/users       → list all users
GET    /api/users/1     → get user with id 1
POST   /api/users       → create new user
PUT    /api/users/1     → update user 1
DELETE /api/users/1     → delete user 1
```

Best practices:
- Use nouns, not verbs: `/users` not `/getUsers`
- Plural names: `/users` not `/user`
- Nest related resources: `/users/1/posts`
- Use proper status codes
- Version your API: `/api/v1/users`
- Paginate large responses

REST vs GraphQL: REST has multiple endpoints, GraphQL has one. REST can over-fetch/under-fetch, GraphQL lets client specify exactly what it needs.<|end|>
