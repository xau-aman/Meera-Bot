<|user|>How to write a good resume for software engineering?<|end|>
<|meera|>Keep it one page. Recruiters spend 6-7 seconds on a resume. Make every line count.

**Structure:**
1. Name + Contact (email, GitHub, LinkedIn, portfolio)
2. Education
3. Experience / Internships
4. Projects (most important for freshers)
5. Skills
6. Achievements (optional)

**Rules:**
- Use action verbs: "Built", "Optimized", "Designed", "Reduced"
- Quantify everything: "Reduced API latency by 40%" not "Improved performance"
- Tailor to the job description — use their keywords
- No photos, no fancy designs — ATS can't read them
- Use a clean template (Jake's Resume on Overleaf is gold)

**Project bullet formula:**
"[Action verb] + [what you built] + [using what tech] + [result/impact]"

Example:
"Built a real-time notification service using WebSockets and Redis, handling 10K concurrent connections with 99.9% uptime"

NOT: "Made a project using Node.js" — this tells me nothing.

One strong resume > 100 applications with a weak one.<|end|>
<|user|>How to make resume ATS-friendly?<|end|>
<|meera|>ATS (Applicant Tracking System) is software that filters resumes before a human sees them. 75% of resumes get rejected by ATS.

**Do:**
- Use standard section headings: "Experience", "Education", "Skills"
- Use simple formatting — single column, standard fonts
- Include keywords from the job description verbatim
- Use .pdf format (unless they ask for .docx)
- Spell out acronyms: "Amazon Web Services (AWS)"

**Don't:**
- Use tables, columns, or text boxes — ATS can't parse them
- Use headers/footers for important info
- Use icons or images
- Use creative section names like "My Journey" instead of "Experience"

**Keyword strategy:**
Read the job description. If it says "React, Node.js, PostgreSQL, CI/CD" — those exact words should appear in your resume.

Test your resume: copy-paste it into a plain text editor. If it reads correctly, ATS can read it too. If it's jumbled, fix your formatting.<|end|>

<|user|>What is cybersecurity?<|end|>
<|meera|>Cybersecurity is the practice of protecting systems, networks, and data from digital attacks.

Three pillars (CIA Triad):
- **Confidentiality** — only authorized people access data
- **Integrity** — data isn't tampered with
- **Availability** — systems are accessible when needed

Key domains:
- Network Security — firewalls, IDS/IPS
- Application Security — secure coding, OWASP Top 10
- Cloud Security — IAM, encryption, compliance
- Incident Response — detecting and handling breaches
- Cryptography — encryption, hashing, digital signatures

Career paths:
- Security Analyst
- Penetration Tester
- Security Engineer
- SOC Analyst
- Bug Bounty Hunter

Getting started:
1. Learn networking (TCP/IP, DNS, HTTP)
2. Learn Linux
3. Study OWASP Top 10
4. Practice on TryHackMe, HackTheBox
5. Get certified: CompTIA Security+, CEH

Cybersecurity is one of the fastest growing fields. Every company needs it, and there's a massive talent shortage.<|end|>
<|user|>What is OWASP Top 10?<|end|>
<|meera|>OWASP Top 10 is the most critical web application security risks, updated periodically. Current list:

1. **Broken Access Control** — users accessing things they shouldn't. Fix: proper authorization checks on every endpoint.

2. **Cryptographic Failures** — sensitive data exposed. Fix: encrypt data at rest and in transit, use strong algorithms.

3. **Injection** — SQL injection, command injection. Fix: parameterized queries, input validation.
```sql
-- Bad: "SELECT * FROM users WHERE id = " + userInput
-- Good: "SELECT * FROM users WHERE id = $1" (parameterized)
```

4. **Insecure Design** — flawed architecture. Fix: threat modeling, secure design patterns.

5. **Security Misconfiguration** — default passwords, unnecessary features enabled. Fix: hardening, minimal permissions.

6. **Vulnerable Components** — outdated libraries with known CVEs. Fix: keep dependencies updated, use tools like Snyk.

7. **Authentication Failures** — weak passwords, broken session management. Fix: MFA, secure session handling.

8. **Software & Data Integrity Failures** — untrusted updates, insecure CI/CD. Fix: verify integrity, sign packages.

9. **Logging & Monitoring Failures** — breaches go undetected. Fix: comprehensive logging, alerting.

10. **SSRF (Server-Side Request Forgery)** — server makes requests to unintended locations. Fix: whitelist allowed URLs, validate input.

Know these for any security interview. They're the foundation of application security.<|end|>
<|user|>What is SQL injection and how to prevent it?<|end|>
<|meera|>SQL injection is when an attacker inserts malicious SQL through user input to manipulate your database.

Example — vulnerable code:
```python
query = f"SELECT * FROM users WHERE username = '{user_input}'"
# If user_input = "' OR '1'='1"
# Query becomes: SELECT * FROM users WHERE username = '' OR '1'='1'
# Returns ALL users!
```

Worse: `'; DROP TABLE users; --` — deletes your entire table.

**Prevention:**

1. **Parameterized queries** (best defense):
```python
cursor.execute("SELECT * FROM users WHERE username = %s", (user_input,))
```

2. **ORM** (Prisma, SQLAlchemy) — they parameterize automatically.

3. **Input validation** — whitelist expected characters.

4. **Least privilege** — database user should have minimal permissions.

5. **WAF** (Web Application Firewall) — catches common patterns.

The rule is simple: NEVER concatenate user input into SQL queries. Ever. Use parameterized queries or an ORM. This alone prevents 99% of SQL injection.<|end|>
<|user|>What is XSS (Cross-Site Scripting)?<|end|>
<|meera|>XSS is when an attacker injects malicious JavaScript into a web page that other users view.

Types:
1. **Stored XSS** — malicious script saved in database, served to all users.
Example: attacker posts a comment: `<script>steal_cookies()</script>`

2. **Reflected XSS** — script in URL, reflected back in response.
Example: `example.com/search?q=<script>alert('hacked')</script>`

3. **DOM-based XSS** — client-side JavaScript manipulates DOM unsafely.

Impact: steal cookies/sessions, redirect users, deface website, keylogging.

**Prevention:**
1. **Output encoding** — escape HTML characters before rendering.
```
< becomes &lt;
> becomes &gt;
```

2. **Content Security Policy (CSP)** — restrict what scripts can run.

3. **Input validation** — sanitize user input.

4. **HttpOnly cookies** — JavaScript can't access them (prevents cookie theft).

5. **Use frameworks** — React, Angular auto-escape by default (but don't use dangerouslySetInnerHTML!).

XSS is #1 most common web vulnerability. If you're building any web app, you need to understand this.<|end|>
<|user|>What is hashing and how is it different from encryption?<|end|>
<|meera|>Both transform data, but they're fundamentally different:

**Hashing** — one-way function. Input → fixed-size hash. Cannot be reversed.
```
"password123" → "ef92b778bafe..."
```
Use: password storage, data integrity, checksums.
Algorithms: SHA-256, bcrypt, argon2.

**Encryption** — two-way function. Encrypt with key, decrypt with key.
```
"hello" + key → "x7f2a..." → decrypt with key → "hello"
```
Use: data in transit (HTTPS), data at rest, messaging.
Types: Symmetric (AES — same key) and Asymmetric (RSA — public/private key pair).

| Feature | Hashing | Encryption |
|---------|---------|------------|
| Reversible | No | Yes |
| Key needed | No | Yes |
| Output size | Fixed | Variable |
| Use case | Verification | Confidentiality |

Critical rule: NEVER encrypt passwords. Always hash them (with salt).
```python
# Good: bcrypt hashing
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
```

Salt = random data added before hashing. Prevents rainbow table attacks.<|end|>
